# Rubric Extraction Eval Playbook

The analysis contract for `docx_v3` rubric extraction. It is the rubric analog of
`P2_EVAL_PLAYBOOK.md`: the suite is the instrument; this file is how you read it
without fooling yourself. If you change extraction behavior and do not run this
suite to a kill criterion, the change is unfalsifiable — do not ship it.

---

## 0. Prime directives (ported from the transcription suite)

1. **The instrument is the prime suspect.** A surprising metric is a scorer bug
   until proven otherwise. The scorer is pinned by `tests/test_scoring.py`
   (known-answer: a perfect extraction must score all-1.0 and PASS; a copy with
   seven injected errors must FAIL with the right per-metric signal). Run it after
   any scorer edit.
2. **Validity before significance.** A truncated or unparseable extraction is
   INVALID, excluded from accuracy aggregates, and fails the gate. Never average a
   number you cannot trust. `finish_reason ∈ {MAX_TOKENS, LENGTH}` ⇒ invalid.
3. **Worst-rubric over mean.** The headline is the worst rubric per metric, not the
   average. A mean of 0.9 across 5 rubrics hides a rubric at 0.5.
4. **One change, one metric, one kill criterion per run.** State the hypothesis, the
   single metric that tests it, and the threshold that kills it, before running.
5. **Read the reports by hand.** The aggregate says IF something moved; the per-scope
   `report_<rubric>.md` says WHERE and suggests the next change. Read at least the
   worst rubric's report every run.
6. **n < 10 across < 2 exams ⇒ provisional.** Every conclusion below that bar is a
   hypothesis, not a result.

## 1. Two-stage attribution (render vs extract)

The pipeline is two stages, scored like P1/P2:

- **RENDER (perception)** — `parser_render.py`: DOCX → markdown. Failure = ink the
  teacher wrote never reaches the LLM.
- **EXTRACT (interpretation)** — LLM: markdown → `RubricExtraction`. Failure =
  content is in the render, mis-structured or dropped.

Every missed GT criterion is auto-attributed from a **single GT**:
- present in the rendered markdown (token-overlap ≥ 0.70) ⇒ `extraction_loss`
  → fix the prompt / model.
- absent from the render ⇒ `render_loss` → fix `parser_render.py`.

`summary.md` reports `render_loss_total` / `extraction_loss_total`. This is the
attribution split that tells you which half to work on. A gold-render mode
(feed hand-cleaned markdown to isolate extraction) is **deferred** — build it only
if render losses turn out to dominate.

## 2. Run modes

- `extract` (the gate) — render → LLM → score vs GT. This is e2e; extraction always
  runs on the real render. `render_audit` runs inside it for free.
- `score_only(predicted, gt, rendered)` — re-score a cached prediction with no
  OpenAI. Used by the self-tests and to re-score a prior run after a scorer change.

```
PYTHONPATH=. python -m tests.rubric_eval_suite.runner --config default --repeats 5
PYTHONPATH=. python tests/rubric_eval_suite/tests/test_scoring.py   # instrument guard
```

## 3. Ground truth format

`benchmarks/<name>.json` is a **type-valid `ExtractRubricResponse`** (it round-trips
through Pydantic; GT that stops validating is itself a schema-drift signal). Authoring
rules:

- Canonical deterministic ids (`q1`, `q1.א`, `q1.א.1`, `q1.c0`). The scorer **ignores
  ids and `index`** — they are generated by the pipeline and shift when structure
  differs. Criteria match by description + points; sub-questions by harmonized label.
- Encode the recursive question tree (depth-2 cap: Question → SubQuestion →
  SubSubQuestion), criteria + sub-criteria, points (exact, 0.25 increments),
  per-node `text` and `example_solution`, and `selection_groups`.
- Encode the **expected `rubric_mismatch` annotations** for any genuine teacher error
  in the source. The extractor passes by reproducing the error + flagging it.
- Fixture set must span: ≥1 selection exam, ≥1 nested-(1)(2) exam, ≥1 bullet/inline-
  scored exam, ≥1 clean-table control (regression guard), ≥1 genuine-teacher-error
  rubric. Missing a mode makes that part of the gate vacuous.

## 4. Scoring semantics

Matching:
- **Questions** by `question_id`. → `question_recall` / `question_precision`.
- **Sub-question tree** recursively by harmonized label (א/a/1 collapse to one index;
  raw-string fallback). `subquestion_structure_match = 1.0` iff the id-tree is
  isomorphic to GT **including leaf-vs-branch classification**. A GT branch matched by
  a predicted leaf (model flattened `(1)/(2)` into criteria) is the **FP2 signal**.
- **Criteria** greedy by `description_ratio ≥ 0.85`; points then feed
  `point_exactness`. **DESCRIPTION-PRIMARY (deviation from the B2 lock — confirm or
  revert):** points are NOT a match key. Requiring points-exact to match would
  double-penalize a pure point error as recall+precision loss and misattribute it;
  matching on description and scoring points separately keeps the signals orthogonal.
  Unmatched GT = missed (recall, FP3); unmatched predicted = spurious (precision).
  A missed GT criterion with 0.55 ≤ ratio < 0.85 to some predicted criterion is
  flagged `merge suspected` (the model fused two GT criteria).
- **Sub-criteria** the same, within matched criteria.

Points: **exact** (tolerance 0). `total_points_correct` = every matched question total
AND rubric total AND `achievable_points` (selection-aware) match GT.

Selection: `selection_match` = group `(choose_k, member-set)` equality.
`achievable_points` = Σ mandatory + Σ over groups of top-`choose_k` totals.

Example solution: **gate-blocking.** GT-non-null ⇒ predicted non-null with ratio ≥ 0.85
(`ok` else `missing`); GT-null ⇒ predicted-null (`ok` else `spurious`).

Annotations: `annotation_match` = predicted reproduces exactly the GT-expected
`rubric_mismatch` annotations (by `(type, target_id)`; Hebrew messages not compared).

Pedagogical mistakes (Step 2c): `pedagogical_match` = predicted `pedagogical_mistakes`
set-equals GT by `(kind, canonical target)`. Targets that are pure question ids are
canonicalized via `canon_q` ('q2'/'2'/'שאלה 2' all match); criterion-style ids compare
verbatim (canon_q digit-concatenation would collide them). GTs with ZERO mistakes are
the detector's false-positive guard — predicting any mistake there fails the gate.
Tier B is an LLM judgment, so treat pedagogical results with the same k-repeats
discipline as any other non-deterministic surface (§6).

**The pedagogical consistency invariant (R2 ruling, 2026-07-10):** GT
`pedagogical_mistakes` ≡ TierA(faithful draft) ∪ expected-Tier-B. Tier A is
deterministic over the draft, so a GT that expects less than Tier A's output on a
faithful draft makes the gate un-passable by construction (the more faithful the
extraction, the more certainly it fails — the bagrut contradiction). Tier-A
expectations are TRANSCRIBED from a probe emission, never hand-authored; Tier-B
kinds (structural_mislabel, orphan_criterion) are the only hand-authored
expectations. Enforced offline by the round-trip test, which runs the REAL Tier A
on every fixture's faithful draft. A teacher point-error legitimately appears on
BOTH surfaces (rubric_mismatch annotation + point_sum_mismatch pedagogical) — two
granularities of one event, per the codebase's annotations+flags pattern.
Corollary: pedagogical_match is a second never-reconcile tripwire — a model that
"fixes" the teacher's sums silences Tier A and fails on the MISSING expected
mistake (PREDICTIONS P-PED1). Authoring rules: GT_AUDIT.md.

Question/sub-question text (PR-1 instrument — **UNGATED-DIAGNOSTIC, all three**):
- `question_text_fidelity_min` / `subquestion_text_fidelity_min` — worst-node
  `text_ratio` over nodes WITH GT text (mean forbidden). Null semantics: GT text
  null ⇒ `None` (not-comparable — null means "no task prose in the document",
  never a 0.0 wall); GT present + predicted null ⇒ 0.0 (a real miss).
- `text_line_recall_min` — worst-node fraction of GT text lines (normalized,
  non-empty) with ≥1 predicted line at ratio ≥ `LINE_TAU` (0.85, provisional).
  `ratio` is length-weighted, so a silently dropped constraint line in a long
  spec still scores ~0.93 (the doc-ratio blindspot); line recall is the omission
  detector, ratio is the fidelity detector — two numbers, two failure classes.
- **NOT in `gate_pass`** (grep gates.py: no text criterion). Calibration path:
  a gate threshold, if ever added, is to be pre-registered from the measured
  distribution of live runs — never guessed. GT text conventions + open-item
  spans: GT_AUDIT.md addendum (PR-1); expected-low-tail nodes: PREDICTIONS.md
  P-T2. The round-trip ceiling (all 1.0/None) is pinned offline in test_fp123.

**Consistency is HEALTH, never gated.** `point_sum_consistency` (Σ children ==
node points, recursive) is reported in every report, but a teacher's rubric may be
legitimately inconsistent — gating on it would punish faithful extraction of a flawed
source (the rubric analog of the rejected transcription Policy 1). The gate criterion
for that case is `annotation_match`, not consistency.

## 5. The conjunctive gate

A rubric PASSES iff ALL hold simultaneously: question recall/precision == 1;
subquestion_structure_match == 1; criterion recall/precision == 1; subcriterion
recall/precision == 1; point_exactness == 1; total_points_correct; selection_match;
example_solution_fidelity == 1; annotation_match; pedagogical_match; valid;
cost ≤ ceiling. Partial
improvement that masks a regression must not pass — that is the point of conjunction.

## 5b. Provenance, cost, and the truncation guard

A result is a function of `(fixtures, config, prompt_version, model_version)` — and
now the record proves it: the pipeline stamps `input_tokens`/`output_tokens`
(accumulated across retries — cost is total spend), the LAST call's `finish_reason`,
and `llm_model` into `ExtractionMetrics` (via `include_raw=True`, the GraderAgent
convention).

Division of responsibility (one concept, one place): the PIPELINE measures tokens;
the CONFIG owns the price table (`price_per_1m_input`/`price_per_1m_output` — prices
drift, so a stale price is a one-line reviewable config diff, never a code change);
the RUNNER multiplies; the GATE judges dollars against `cost_ceiling`.

`finish_reason ∈ {length, max_tokens}` ⇒ the record is INVALID (truncation guard,
live as of this revision) and is excluded from accuracy aggregates — validity before
significance.

Failure isolation: one doc's transport error becomes an INVALID record with the
exception as `invalid_reason`; the suite run always completes (worst-doc discipline —
a crashing doc is signal, and losing the run would destroy it).

Model switching: one config file = one (model, provider, prices, knobs) pairing.
`_llm_params` owns per-family constructor policy (reasoning family: no temperature,
effort knob, bigger budget). Never switch via raw env — the cost computation would
silently use the wrong prices.

---

## 6. Determinism

gpt-4o at temp 0 is **not** guaranteed deterministic. Before trusting any single run:
probe the control rubric at `--repeats 5` and inspect per-metric variance. If
non-deterministic, evaluate every change at `--repeats N` pre/post and compare worst
runs, not single runs (the P2 lesson: single-run comparisons on non-deterministic
models are uninterpretable). One variable per run regardless.

---

## 7. Per-FP kill criteria (the pending pipeline work)

When the FP fixes land (one at a time, measured against the baseline):

- **FP1 (selection):** `selection_match` and `total_points_correct` reach 1.0 on the
  selection fixture(s) with **zero regression** on `selection_match` for non-selection
  rubrics (no spurious groups invented).
- **FP2 (nesting):** `subquestion_structure_match` reaches 1.0 on the nested fixture(s)
  with **zero new `leaf_vs_branch`** on any previously-isomorphic rubric.
- **FP3 (bullet/inline criteria):** `criterion_recall` rises on the bullet fixture(s)
  with **`criterion_precision` staying 1.0 on every already-clean question** — not one
  spurious criterion. This is the precision kill criterion; it is the whole reason the
  fix is gated.

If a fix lifts its target metric but trips another rubric's gate, it has not shipped —
the conjunctive gate is the arbiter.



## §11. RUNLOG.md — the run ledger (hard memory)

`tests/rubric_eval_suite/RUNLOG.md` is the append-only ledger of every run,
analysis, and experiment-variable change. results.json stores what happened;
the RUNLOG stores what it MEANT and what was DECIDED — the two things that
otherwise live in chat context and die with it.

### Rules
1. **Append-only.** Never edit or delete a past entry. A wrong entry is
   corrected by a new entry that names it (the retraction discipline).
2. **An analysis is not DONE until its RUNLOG entry exists.** This extends the
   §9 done-criteria. No new run is planned before the previous run's entry is
   written — next-run planning reads the ledger, not memory.
3. **Every experiment-variable change gets an entry, run or no run.** Editing a
   config (model, effort, ceiling, budgets), the prompt (bump
   EXTRACTION_PROMPT_VERSION), GT benchmarks, or scorer/gate logic = a
   VARIABLE-CHANGE entry: what, why, who, and which future comparison it
   affects. (Origin: reasoning_effort drifted high→medium with no author and
   no record; a unit test caught it by accident. The ledger is the deliberate
   version of that tripwire.) **A convention RULING is such a change even when
   no file's variable moves, and it must be logged AS a ruling — not folded
   silently into the GT-data entry that enacts it, and regardless of which
   thread does it. Origin: the two-column convention was enacted in GT+prompt
   (CHANGE 2026-07-09) but never recorded as a ruling, forcing the 2026-07-11
   PR-2 reconciliation forensics — the named cost of skipping this.**
4. **Entries are ≤ ~20 lines.** The full analysis lives in its own report; the
   entry links to it. The ledger is for scanning, not reading.
5. **One entry per analyzed run**, including failed/aborted/smoke runs (marked
   as such — an invalid run is signal).

### Run-entry template
    ## RUN <results-folder-id>
    ref: <run-id this compares against, or "none (first baseline)">
    purpose: <baseline | instrument-verification probe | variable test: <X>>
    variable changed vs ref: <exactly one, or "none">
    config: <name>  k: <n>  model: <id>  effort: <val>
    prompt_version: <stamped>  suite_hash: <stamped>
    validity: <n>/<n> valid; excluded: <doc: reason, ...>
    gate: <x>/5 worst-run pass
    predictions: <P-id>: CONFIRMED | FALSIFIED | UNDECIDABLE(<why>) ...
    deltas vs ref: <only what CHANGED, per fixture-criterion; no full dumps>
    surprises: <new failure classes + attribution: instrument|GT|spec|model>
    decisions: <GT edits / rulings / config changes taken, with links>
    next: <one action> — kill criterion: <pre-registered>
    cost: $<total> (<mean>/doc)  wall: <t>
    corrections: <retractions of prior entries, or "none">

### Variable-change entry template
    ## CHANGE <date>
    what: <file/field, old → new>
    why: <one line>  by: <who decided>
    affects: <which run comparisons this invalidates or requires re-baselining>

### What does NOT go in the ledger
Raw metric dumps (results.json has them), speculation without a run or a
desk-check behind it, and anything already recorded verbatim in GT_AUDIT.md or
PREDICTIONS.md — link, don't duplicate.
